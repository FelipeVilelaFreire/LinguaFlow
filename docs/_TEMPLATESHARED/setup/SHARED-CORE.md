# Shared-Core вЂ” The Single Brain

> The package every platform consumes. Types, hooks, services, constants, theme вЂ” all live here.
> Mobile and web are render-only вЂ” they call `useXxxScreen()` from shared-core and display the result.

Stack: **TypeScript 5 В· pure modules В· no React Native, no Next.js imports** (must run in both).

---

## вљЎ The single principle

```
shared-core = BRAIN     в†’  types В· hooks В· services В· constants В· theme В· validation В· mappers В· strings
mobile      = RENDER    в†’  React Native JSX. Zero logic.
web         = RENDER    в†’  Next.js JSX. Zero logic.
```

If you find logic in `mobile/` or `web/` that isn't pure render, it belongs in `shared-core/`.

For shared features, keep the web and mobile trees intentionally comparable. UI can adapt by platform, but feature names, responsibilities, hook ownership, strings, tokens, and domain types should line up. Use `docs/PARITY-AUDIT.md` and update `docs/PARITY-AUDIT-EXECUTION.md` when reviewing a folder pair.

## рџљ« What CANNOT live in shared-core

- React Native imports (`react-native`, Expo APIs)
- Next.js imports (`next/navigation`, `next/link`)
- DOM APIs (`document`, `window`, `localStorage`)
- Node-specific modules (`fs`, `path`)
- Platform-specific rendering primitives

Anything that touches a platform-specific API gets **injected** via a wrapper (see Storage Adapter and API Client below).

## рџЄќ Hook Patterns вЂ” same shape as `_TEMPLATENEXTDJANGO`

A screen hook in shared-core looks identical to a Next.js hook, but it must be platform-neutral:

```typescript
// packages/shared-core/src/hooks/{{entity}}/use{{Entity}}Screen.ts
import { useEffect, useState, useCallback } from 'react';
import { {{entity}}Service } from '../../services/{{entity}}Service';
import { getErrorMessage } from '../../utils/errors';
import type { {{Entity}}, {{Entity}}UpdateData } from '../../types/{{entity}}';

export interface Use{{Entity}}ScreenOptions {
  id: number;
  onSuccess?: () => void;
  onError?: (msg: string) => void;
}

export interface Use{{Entity}}ScreenReturn {
  data: {{Entity}} | null;
  loading: boolean;
  error: string | null;
  isSaving: boolean;
  handleSave: (payload: {{Entity}}UpdateData) => Promise<void>;
  handleDelete: () => Promise<void>;
  refresh: () => Promise<void>;
}

export function use{{Entity}}Screen({ id, onSuccess, onError }: Use{{Entity}}ScreenOptions): Use{{Entity}}ScreenReturn {
  /* same body as the web-only version вЂ” only imports differ */
}
```

Platform-specific concerns (router calls, Alert.alert vs window.alert, animations) **stay in the platform layer** and use `onSuccess` / `onError` callbacks from the hook.

## рџ”Њ Platform Adapters

Two things shared-core cannot know natively: storage and HTTP.

### Storage Adapter

```typescript
// packages/shared-core/src/lib/storage.ts
export interface StorageAdapter {
  getItem(key: string): Promise<string | null>;
  setItem(key: string, value: string): Promise<void>;
  removeItem(key: string): Promise<void>;
  multiRemove(keys: string[]): Promise<void>;
}

let adapter: StorageAdapter | null = null;

export function setStorageAdapter(a: StorageAdapter) { adapter = a; }
export function storage(): StorageAdapter {
  if (!adapter) throw new Error('Storage adapter not set. Call setStorageAdapter() at app boot.');
  return adapter;
}
```

**Mobile sets it** with `AsyncStorage` from `@react-native-async-storage/async-storage`.
**Web sets it** with a `localStorage` shim returning Promises.

### API Client

```typescript
// packages/shared-core/src/lib/apiClient.ts
export interface ApiClient {
  get<T>(path: string, params?: Record<string, unknown>): Promise<T>;
  post<T>(path: string, body: unknown): Promise<T>;
  patch<T>(path: string, body: unknown): Promise<T>;
  delete(path: string): Promise<void>;
}

let client: ApiClient | null = null;

export function setApiClient(c: ApiClient) { client = c; }
export function api(): ApiClient {
  if (!client) throw new Error('API client not set. Call setApiClient() at app boot.');
  return client;
}
```

Both platforms wrap axios (with platform-specific token reading) and call `setApiClient` at boot.

## рџЋЁ Design Tokens вЂ” platform-neutral

Tokens are plain numbers/strings in shared-core, then **rendered differently per platform**:

```typescript
// packages/shared-core/src/theme/spacing.ts
export const SPACING = {
  xs: 4, sm: 8, md: 12, lg: 16, xl: 20, '2xl': 24, '3xl': 32,
} as const;
```

```typescript
// packages/shared-core/src/theme/colors.ts
export const COLORS = {
  primary: '#0066ff',
  white: '#ffffff',
  gray50: '#fafafa',
  gray100: '#f4f4f5',
  /* ... */
  gray900: '#18181b',
  success: '#10b981',
  warning: '#f59e0b',
  error: '#ef4444',
} as const;
```

**Mobile consumes them** as numeric/string values in StyleSheet:
```typescript
import { SPACING, COLORS } from '@{{project}}/shared-core';
const styles = StyleSheet.create({
  card: { padding: SPACING.lg, backgroundColor: COLORS.white },
});
```

**Web consumes them** indirectly вЂ” `web/src/theme/globals.css` writes the same values as CSS Variables, mirroring shared-core. (See `DESIGN.md`.)

Both must stay in sync. If you change `COLORS.primary` in shared-core, also update `--color-primary` in `web/src/theme/globals.css`.

## рџЊђ Strings (i18n)

```typescript
// packages/shared-core/src/constants/strings/pt-BR.ts
export const PT_BR = {
  actions: { save: 'Salvar', cancel: 'Cancelar', /* ... */ },
  auth: { /* ... */ },
  {{entity}}: { /* ... */ },
} as const;
```

```typescript
// packages/shared-core/src/constants/strings/index.ts
import { PT_BR } from './pt-BR';
import { EN } from './en';
import { DE_DE } from './de-DE';

type Locale = 'pt-BR' | 'en' | 'de-DE';

let currentLocale: Locale = 'pt-BR';

export function setLocale(locale: Locale) { currentLocale = locale; }
export const STRINGS = new Proxy({} as typeof PT_BR, {
  get(_t, prop: string) {
    const dict = currentLocale === 'en' ? EN : currentLocale === 'de-DE' ? DE_DE : PT_BR;
    return dict[prop as keyof typeof PT_BR];
  },
});
```

Both platforms import `STRINGS` from `@{{project}}/shared-core` and use it identically.

Admin labels live under `STRINGS.admin.*` and follow the same three-locale rule. Do not put visible copy in admin `config.jsx`, `constants.js`, table columns, filters, tabs, toasts, placeholders, modals, aria labels, empty states, or errors.

## Web/Mobile Parity Contract

Shared-core owns the contract. Web and mobile own rendering.

When a feature exists on both platforms:

- the domain hook lives once in `packages/shared-core/src/hooks/{domain}/`;
- the service lives once in `packages/shared-core/src/services/`;
- domain types live once in `packages/shared-core/src/types/{domain}/`;
- strings live once in `packages/shared-core/src/constants/strings/`;
- platform hooks are thin wrappers that inject router/Alert/toast behavior;
- component trees are conceptually mirrored (`features`, `cards`, `modals`, `shared`, `ui`);
- differences are recorded in `docs/PARITY-AUDIT-EXECUTION.md`.

The audit starts with `rg --files` on the matching web/mobile folders, then verifies render-only, STRINGS, tokens, and shared-core ownership pair by pair.

## рџЄќ The 5 Hook Levels (same taxonomy as `_TEMPLATENEXTDJANGO`)

| Level | Suffix | Lives in | Use case |
|-------|--------|----------|----------|
| Screen | `useXxxScreen` | `shared-core/hooks/{entity}/` | Fetch + save + delete for a screen |
| Form | `useXxxForm` | `shared-core/hooks/{entity}/` | Form state, validation, transformation |
| Tab | `useXxxTab` | `shared-core/hooks/{entity}/` | Filterable list inside a tab |
| Data | `useXxxData` | `shared-core/hooks/{entity}/` | Just loads data |
| Card | `useXxxCardData` | `shared-core/hooks/{entity}/` | Transforms data for card display |

Plus a sixth: **platform wrappers** (live in the platform, not shared-core):

```typescript
// mobile/src/hooks/{{entity}}/useXxxScreen.ts
import { useXxxScreen as useCore } from '@{{project}}/shared-core';
import { router } from 'expo-router';
import { Alert } from 'react-native';
import { ROUTES } from '@{{project}}/shared-core';

export function useXxxScreen(id: number) {
  return useCore({
    id,
    onSuccess: () => router.back(),
    onError: (msg) => Alert.alert('Erro', msg),
  });
}
```

```typescript
// web/src/hooks/{{entity}}/useXxxScreen.ts
import { useXxxScreen as useCore } from '@{{project}}/shared-core';
import { useRouter } from 'next/navigation';
import { ROUTES } from '@{{project}}/shared-core';

export function useXxxScreen(id: number) {
  const router = useRouter();
  return useCore({
    id,
    onSuccess: () => router.back(),
    onError: (msg) => window.alert(msg),
  });
}
```

The wrapper is a 5-line file. All real logic stays in shared-core.

## рџљ¦ Pre-merge Shared-Core Checklist

- [ ] No imports from `react-native`, `next`, DOM globals, or Node-only modules?
- [ ] Every hook receives `onSuccess` / `onError` callbacks instead of calling router/Alert directly?
- [ ] Every type is exported from `src/index.ts`?
- [ ] `dist/` rebuilt after changes (`yarn build` in `packages/shared-core/`)?
- [ ] Both platforms still type-check after the change (`tsc --noEmit` in `mobile/` and `web/`)?
- [ ] New strings added to `pt-BR.ts`, `en.ts`, and `de-DE.ts`?
- [ ] Admin strings use `STRINGS.admin.*` instead of literals in config/constants/JSX?
- [ ] Shared feature trees are mirrored or differences are recorded in `docs/PARITY-AUDIT-EXECUTION.md`?
- [ ] Service uses the injected `api()` client, never `fetch` or `axios` directly?
